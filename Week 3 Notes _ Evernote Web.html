Week 3 Notes
More About Functions

  • Parameters are the argument placeholders in the function definition (didn't know that...)
  • Okay, so if an argument is defined, you can use the line of code "a = a || 0;"
    • For numbers, 0 is truthy...? i don't habeeb it.  That's cuz it's not.
    • The assignment that he's talking about is 'short circuit evaluation'; the last value is always assigned.
  • Typeof operator returns a string
  • Ternary Operators
    • I know how they work
  • Variables within functions are 'lexically scoped'
    • Local to the function
  • Garbage Collection
    • When you define a variable in JS, it has a scope
    • As soon as the program moves outside of that scope, it gets garbage collected, and is gone forever
  • Closure
              var counterConstructor = function() {
                 var start = 0;
                 var counter = function() {
                     return start++;
                  };
                return counter;
              }
  • So counterConstructor needs the start variable -- it has closure over the start variable; it doesn't get garbage collected
    • But nothing has actually been invoked yet; just referenced
    • And like, if you reference / invoke this multiple times, is it pointing at the same reference or are they different references?
    • So they both have closure over a variable called 'counter', but they're referring to different values
    • So you use this for this namespacing
    • Closure isn't so much something you utilize, as much as it is something that happens
    • Basically, if you've got a reference to a function, that also uses a variable, that variable doesn't get garbage collected.  This is a little bit weird, and I need to make an effort to use it extra plus plus more, so that I get familiar with it 
    • To manually garbage collect something, just set a val to null
Pair Programming
  • So have one driver, one navigator -- driver types, navigator is there to bounce 
Functional programming
  • In JS, functions are 'first-class citizens'
    • They can go anywhere
      • So you can pass one function into another function
      • Or define a function inside of another function
      • or return a function from another function
    • So Anything you can do with any value, you can do with a function
      • (Not true of languages like Java or C++ where you can't really do this so much)
  • Compare with imperative programming, where you write a series of commands, one after another, to manipulate the state of the program
  • functional is more declarative -- try to avoid manipulating state
  • Composable
    • Decompose problems into smaller and smaller pieces -- and then make the small pieces composable, and put them back together to solve large problems
    • The way that you can make functions composable is to pass functions into other functions, etc
    • COMPOSABLE
  • You want to avoid manipulating state DIRECTLY
    • So if you're constantly manipulating the state, it's possible for the data to get out of sync with what you think it should be
    • So instead of manipulating old data, you just overwrite that data with new data -- and then you know that it's fresh and clean
    • So you sort of treat the var's like they're immutable -- and when you want new data, you just overwrite that
  • To this is a pure function -- takes input, returns output, but doesn't have any side effects. Doesn't manipulate variables outside of its scope. If you're using a functional style, you should have mostly functional programs.
var changeName = function(name) {
    var newUser = {name : name}
    return newUser
}
  • If possible, you want it to have all of the variables that it needs as arguments.  It's not terrible to have it read global variables, but it's best if you give it everything as an argument -- it also makes debugging easier
  • But of course, there are limitations.
    • It's easier to make the smaller blocks functional
    • But people are interacting with website, and want their changes to be saved.  So it's fine for user changes to modify state.
    •  Functional programming is a tool; use it as such
    • And functional programming isn't super appropriate for event handling
  • doStuff is a callback function.  I never call it myself, JS calls it for me when the button is clicked.  Because I can't control what is done with the return value, event handlers need to have side effects in order to do productive things.  The return value doesn't go anywhere.
<button ng-click="doStuff()">button</button>
$scope.doStuff = function(){}

Code Examples:
var words = ['this', 'is', 'a', 'string'];

var capOneWord = function(oneWord) {
    return oneWord.toUpperCase()
}

var capWords = function(wordsArray) {
    var output = [];
    for (var i = 0; i < wordsArray.length; i++) {
        output[i] = capOneWord(wordsArray[i])
    }
    return output;
}

var capitalizedWords = capWords(words);

console.log(capitalizedWords);
  • Beaks the problem down into smaller steps
  • BUT we don't like for loops -- for loops are very imperative.  Like, do this thing, and then do this other thing...
    • Plus, you're directly manipulating the output var (though granted you just created it for the purpose of manipulating it)
  • So, an even more functional way to do this:
var capWords = function(wordsArray) {
    var output = []
    wordsArray.forEach(function(cur,i,arr){ output.push( capOneWord( element ) ) } );
}
  • Oh okay, that's cool -- forEach is exactly how I would have done that myself.  And it's the more functional way to do that!  Yay.  You know, reduce might be an even better way to do that. (In fact, map might be an even more functional way to do that)
    • Aids in code reuse
  • Most common to use an anonymous function as a callback, but if you find yourself using the same callback over and over, it would make sense to name it
  • Even more functional:
var capWords = function(wordsArray) {
     return wordsArray.map(function(cur,i,arr){
        return capOneWord(cur);
     })
}
  • map
    • 'Map is a functional programmers best friend'
    • I may very well be able to totally replace for-loops with map
      • Or, in my case, forEach
    • Map is more functional -- returns a new array instead of modifying an old array.
      • Obviously in the callback you need to be returning a value
  • So you can break this problem down into three very small, one line functions
    • Smaller, tighter code
    • Smaller units means that it's easier to debug
  • Man, it's really going to be a challenge to refactor my infinite scroll app in this way.
    • Eh, fuck that noise, dude.  I don't really want to.  I don't think it's very necessary.
  • In regards to providing arguments to built in methods, it makes sense not to pass arguments that you're not going to use, so that someone else reading your code can see that those arguments aren't used
So Imagine a function that is checking to see if strings have two or more 'b's in them (to get in to the 'B Club')

var people = [
    'Alice Bobbits',
    'Bob Baggins',
]

... Basically, he's going over the filter array.
  • Filter returns a new array of a different length, where only items that have returned true are included.
  • Reduce
    • Turns an array into a single value
    • array.reduce(function(runningTotal, element {}, initialValue)
    • For the first time it;s run, runningTotal is initial value
    • return positive, negative, or 0 (return 0 if they're the same and you dgaf about the order)
  • Sort
    • A destructive function -- modifies the original array
    • Works well with objects
    • Wow, raphael's story about interviewing for a job and getting the job before he got home b/c he knew how sort works

Check out MomentJS at some point
